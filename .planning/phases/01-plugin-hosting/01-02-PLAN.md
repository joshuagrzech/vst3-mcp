  <action>
Create `src/bin/integration_test.rs` -- a binary that exercises ALL Phase 1 success criteria against real VST3 plugins installed on the system. This is NOT a unit test (those exist already); this is a manual verification tool run against real plugins.

Add the binary target to `Cargo.toml`:
```toml
[[bin]]
name = "agent-audio-integration-test"
path = "src/bin/integration_test.rs"
```

The binary should accept optional CLI arguments:
- No arguments: scan default paths, test with whatever plugins are found
- `--path <dir>`: scan a specific directory
- `--class-id <hex>`: test a specific plugin by classId (skip scanning)
- `--cycles <N>`: number of load/unload cycles for teardown testing (default: 10)

The binary should run these test phases in order, printing clear PASS/FAIL results:

**Phase A: Plugin Discovery (HOST-01)**
1. Call `scan_plugins()` (or `scan_plugins_safe()` with scanner binary if available)
2. Print discovered plugins: classId, name, vendor, category, path
3. PASS if at least 1 plugin discovered. FAIL if empty.

**Phase B: Plugin Loading (HOST-02)**
For each discovered plugin (or the specified classId):
1. Load the module via `Arc::new(VstModule::load(path))`
2. Parse the hex classId back to a TUID (add a `hex_string_to_tuid()` helper if needed)
3. Create `PluginInstance::from_factory(module, &class_id, host_app, handler)`
4. Print plugin name and vendor from the scan info
5. PASS if instance created without error. FAIL with error message.

**Phase C: Full Lifecycle (HOST-03)**
For each loaded plugin:
1. Assert state is `Created`
2. Call `setup(44100.0, 512)`, assert state is `SetupDone`
3. Call `activate()`, assert state is `Active`
4. Call `start_processing()`, assert state is `Processing`
5. Process a small block of silence (2 channels, 512 samples)
6. Call `stop_processing()`, assert state is `Active`
7. Call `deactivate()`, assert state is `SetupDone`
8. Print bus info and parameter count for informational purposes
9. PASS if all transitions succeed. FAIL with specific transition that failed.

**Phase D: Teardown Stress Test (HOST-04)**
For each plugin, run N load/unload cycles:
1. Load module, create instance, run full lifecycle (setup -> activate -> start_processing -> process -> stop -> deactivate -> drop)
2. Drop the instance, then drop the module (Arc should handle this)
3. Print cycle number on each completion
4. PASS if all N cycles complete. FAIL with cycle number and error.

**Phase E: Unified vs Split Detection (HOST-05)**
For each tested plugin:
1. After loading, check if the controller was obtained via cast (unified) or via separate factory creation (split)
2. This requires adding a `pub fn is_controller_separate(&self) -> bool` method to PluginInstance that checks whether the controller pointer differs from the component cast
3. Print whether each plugin uses unified or split component/controller
4. Report a summary: how many unified, how many split

Exit with code 0 if all phases pass for all plugins tested. Exit with code 1 if any failure.

Use `tracing` for debug output and plain `println!` for test results (structured, easy to read).

Initialize tracing with env-filter so `RUST_LOG=debug` enables verbose COM-level logging.
