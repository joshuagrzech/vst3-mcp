<objective>
Implement Linux::IRunLoop and XEmbed protocol so plugin UIs are properly embedded and responsive.

Purpose: Plan 01 creates the window and attaches the plugin view, but without IRunLoop and XEmbed, the plugin's UI won't be interactive. Many plugins (Qt/GTK/VSTGUI-based) require IRunLoop to register their X11 file descriptors and timers for event processing. XEmbed protocol ensures the plugin's child window is correctly embedded instead of appearing as a floating window.

Output: HostRunLoop COM implementation with FD/timer dispatch integrated into winit event loop, XEmbed protocol helpers, complete embedding handshake in EditorWindow.
</objective>

Create src/gui/runloop.rs implementing HostRunLoop (Pattern 3 from research):

HostRunLoop struct:
- event_handlers: Mutex&lt;HashMap&lt;RawFd, ComPtr&lt;IEventHandler&gt;&gt;&gt;
- timers: Mutex&lt;Vec&lt;TimerEntry&gt;&gt;

TimerEntry struct:
- handler: ComPtr&lt;ITimerHandler&gt;
- interval_ms: u64
- last_fire: std::time::Instant

impl vst3::Class for HostRunLoop with type Interfaces = (Linux::IRunLoop,).

impl Linux::IRunLoopTrait for HostRunLoop:
- registerEventHandler(handler, fd): Create ComPtr::from_raw(handler), insert into event_handlers map with fd as key, return kResultOk, log "Registered event handler for FD {fd}"
- unregisterEventHandler(handler): Retain event_handlers where handler pointer doesn't match, return kResultOk, log "Unregistered event handler"
- registerTimer(handler, interval_ms): Create ComPtr::from_raw(handler), push TimerEntry to timers vec with last_fire = Instant::now(), return kResultOk, log "Registered timer with interval {interval_ms}ms"
- unregisterTimer(handler): Retain timers where handler pointer doesn't match, return kResultOk, log "Unregistered timer"

Add public methods for event loop integration:
- pub fn dispatch_ready_fds(&amp;self, ready_fds: &amp;[RawFd]): Lock event_handlers, for each ready FD call handler.onFDIsSet(fd)
- pub fn dispatch_timers(&amp;self): Lock timers, for each timer check if elapsed &gt;= interval_ms, if so call handler.onTimer() and update last_fire

Use ComWrapper::new(HostRunLoop { event_handlers: ..., timers: ... }) pattern from param_changes.rs.

Update src/gui/mod.rs to export HostRunLoop from runloop module.

Follow research Example 3 implementation exactly. Use tracing::debug! for FD/timer registration logs.

Update EditorWindow::new() to initialize XEmbed:
1. Create HostRunLoop: let runloop = Arc::new(ComWrapper::new(HostRunLoop::new()));
2. Pass runloop to PlugFrame::new(runloop.clone())
3. After winit window creation, establish X11 connection: x11rb::connect(None)? for direct X11 access
4. Extract screen from connection for root window access
5. After plug_view.attached(), listen for CreateNotify events by selecting SUBSTRUCTURE_NOTIFY on parent window (use x11rb::protocol::xproto::change_window_attributes with event_mask)
6. Store x11_connection and runloop in EditorWindow

Update open_editor_window() event loop to handle XEmbed and dispatch:

In event_loop.run() handler, add match on event:
- Event::NewEvents(StartCause::Init): Poll X11 connection for CreateNotify
- Event::AboutToWait:
  1. Poll X11 connection for CreateNotify events (use x11rb connection.poll_for_event())
  2. If CreateNotify received and create.parent == parent_window_id:
     - Store plugin_window = create.window in EditorWindow.plugin_window_id
     - Call send_xembed_message(conn, plugin_window, XEMBED_EMBEDDED_NOTIFY, 0, timestamp, 0)
     - Log "Plugin child window detected, sent XEMBED_EMBEDDED_NOTIFY"
  3. Call runloop.dispatch_timers()
  4. If poller exists and has FDs registered, call poller.wait() with 0 timeout, then runloop.dispatch_ready_fds()
- Event::WindowEvent { event: WindowEvent::CloseRequested, .. }: target.exit()

Note: This integrates Pattern 5 (child window detection) and dispatches IRunLoop events per Pattern 3. The poller.wait() approach is simpler than full mio integration for Phase 04.1 (research Open Question 5).

Initialize poller field in EditorWindow::new():
- Create polling::Poller::new()?
- Register FDs dynamically when runloop.registerEventHandler is called (this requires HostRunLoop to notify EditorWindow -- deferred complexity, leave poller as None for Phase 04.1, document TODO)

For Phase 04.1 scope constraint: Set poller = None initially. Timers will dispatch (sufficient for many plugins). FD polling can be added in follow-up if specific plugins need it. Document this as a known limitation.
