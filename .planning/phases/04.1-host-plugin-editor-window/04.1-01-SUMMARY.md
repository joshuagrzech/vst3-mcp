# Phase 04.1 Plan 01+02: Plugin Editor Window Hosting Summary

**One-liner:** GUI module with winit window, IPlugView lifecycle, IRunLoop event dispatch, XEmbed protocol, and open_editor MCP tool for hosting VST3 plugin editor windows on Linux X11.

## What Was Built

### GUI Module Structure (src/gui/)

Five new files implementing complete VST3 editor window hosting:

1. **src/gui/mod.rs** - Module re-exports
2. **src/gui/window.rs** - EditorWindow with winit integration
3. **src/gui/plugframe.rs** - IPlugFrame + IRunLoop COM implementation
4. **src/gui/runloop.rs** - Linux::IRunLoop with FD and timer management
5. **src/gui/xembed.rs** - XEmbed protocol helpers

### EditorWindow (src/gui/window.rs)

Complete IPlugView lifecycle management:
- `IEditController::createView("editor")` to get IPlugView
- `IPlugView::isPlatformTypeSupported("X11EmbedWindowID")` validation
- `IPlugView::getSize()` for preferred window dimensions
- `IPlugView::setFrame()` with combined IPlugFrame+IRunLoop
- `IPlugView::attached()` with X11 Window ID
- `IPlugView::removed()` + `setFrame(null)` in Drop for clean teardown

Event loop integration:
- CreateNotify polling for child window detection
- XEmbed EMBEDDED_NOTIFY handshake on child window creation
- IRunLoop timer dispatch on each AboutToWait event
- FD polling via polling crate for registered event handlers
- Focus/activation XEmbed messages on window focus changes

### IPlugFrame + IRunLoop (src/gui/plugframe.rs)

Combined COM object implementing both IPlugFrame and Linux::IRunLoop:
- `resizeView()` accepts all resize requests (logs dimensions)
- IRunLoop methods delegate to shared HostRunLoop instance
- Plugins obtain IRunLoop via queryInterface on the IPlugFrame pointer

### HostRunLoop (src/gui/runloop.rs)

Linux::IRunLoop COM implementation:
- `registerEventHandler(handler, fd)` - stores FD → handler mapping
- `unregisterEventHandler(handler)` - removes by pointer comparison
- `registerTimer(handler, interval_ms)` - stores timer with fire tracking
- `unregisterTimer(handler)` - removes by pointer comparison
- `dispatch_ready_fds(fds)` - calls `onFDIsSet()` for ready FDs
- `dispatch_timers()` - fires expired timers via `onTimer()`
- Proper COM reference counting (AddRef on register, Release on unregister)
- Clean Drop implementation releases all held references

### XEmbed Protocol (src/gui/xembed.rs)

XEmbed helpers using x11rb:
- XEmbed atom interning via `atom_manager!` macro
- `send_xembed_message()` - generic ClientMessage sender
- `send_embedded_notify()` - XEMBED_EMBEDDED_NOTIFY (handshake completion)
- `send_window_activate/deactivate()` - focus coordination
- `send_focus_in/out()` - keyboard focus management

### open_editor MCP Tool (src/server.rs)

New MCP tool added to AudioHost:
- Verifies plugin is loaded, gets plugin name for title
- Spawns dedicated GUI thread (std::thread::spawn)
- Calls `gui::open_editor_window()` on GUI thread
- Blocks MCP call until window closes via mpsc channel
- Returns JSON: `{"status": "closed", "message": "Editor window closed"}`

## Key Design Decisions

1. **Combined IPlugFrame + IRunLoop COM object** - Plugins cast IPlugFrame (from setFrame) to IRunLoop via queryInterface. By listing both interfaces in `type Interfaces`, the vst3 crate auto-generates correct queryInterface vtable.

2. **Dedicated GUI thread** - Winit event loop is blocking and cannot run inside Tokio's async runtime. `std::thread::spawn` creates OS thread for winit event loop. MCP tool blocks via mpsc channel until GUI thread exits.

3. **Raw COM pointer management** - IRunLoop stores raw `*mut IEventHandler` and `*mut ITimerHandler` with manual AddRef/Release, matching the C++ VST3 SDK pattern. This avoids ComPtr ownership issues with HashMap/Vec storage.

4. **Fixed-size window** - Phase 04.1 uses non-resizable window (avoids resize loop pitfall). The plugin's preferred size from getSize() is used directly.

5. **FD polling integration** - Uses the `polling` crate (same foundation as Tokio's mio) to monitor file descriptors registered by plugins via IRunLoop. Polls on each event loop iteration with zero timeout (non-blocking).

## Known Limitations

1. **Fixed-size window** - No resize support (deferred to avoid resize loop pitfall)
2. **Linux X11 only** - Wayland and other platforms not supported yet
3. **open_editor blocks** - MCP tool call blocks until window closes (intentional for Phase 04.1)
4. **No visual verification in CI** - Editor window tests require a display server
