# Phase 04.1: Host Plugin Editor Window - Research

**Researched:** 2026-02-15
**Domain:** VST3 IPlugView editor window embedding on Linux X11 with native windowing in Rust
**Confidence:** MEDIUM-HIGH

## Summary

Phase 04.1 brings forward GUI-03 (deferred from v2) as urgent work: the host must display the plugin's editor window so users can see and interact with the plugin's GUI while the AI controls parameters via MCP during production sessions. This is critical for the offline MVP to be usable in real music production workflows.

The core technical challenge is embedding a child VST3 plugin's IPlugView (which creates its own X11 window) into a parent window created and managed by the Rust host application. On Linux, this requires:
1. Creating a parent X11 window using a Rust windowing library (winit or x11rb)
2. Calling `IEditController::createView()` to get the plugin's IPlugView COM interface
3. Calling `IPlugView::attached()` with the X11 Window ID (platform type `kPlatformTypeX11EmbedWindowID`)
4. Implementing the XEmbed protocol handshake (XEMBED_EMBEDDED_NOTIFY, focus, activation messages)
5. Implementing `Linux::IRunLoop` interface so the plugin can register event handlers and timers for its UI thread
6. Handling resize events via `IPlugView::onSize()` and window close/detach via `IPlugView::removed()`

The Steinberg VST3 SDK provides a complete reference implementation in C++ (`editorhost` sample), which shows the exact X11 API calls and XEmbed protocol messages required. Rust has two viable paths: (1) use **winit** (mature, cross-platform, supports `with_parent_window` on X11) with manual XEmbed and IRunLoop implementation, or (2) use **x11rb** (pure Rust X11 bindings) with full manual control over window creation and event handling. The **baseview** crate (RustAudio project) is designed specifically for audio plugin UIs with VST/CLAP in mind, but is primarily for plugin-side UI (not host-side embedding), though its patterns may be instructive.

**Primary recommendation:** Use **winit 0.30+** for window creation (cross-platform, well-maintained, integrates with Tokio via event loop), implement XEmbed protocol manually using **x11rb** for X11-specific atom and message operations, and provide a minimal `Linux::IRunLoop` COM implementation using the existing `ComWrapper` pattern from the codebase. This is an additive feature -- the MCP server continues running on stdio, and the editor window runs on a separate OS thread with its own event loop.

**Critical scope constraint:** This is an offline MVP feature. The window runs as a standalone native application, NOT embedded in a DAW. DAW plugin embedding (Phase 7+) will require significant architectural changes (stdio transport won't work, different threading model, host-provided parent window). Phase 04.1 focuses solely on "standalone host shows plugin GUI."

### Pattern 1: VST3 Editor Lifecycle (IPlugView Sequence)

**What:** The host must follow a strict sequence when opening and closing a plugin's editor window. This is independent of the audio processing lifecycle (Created -> SetupDone -> Active -> Processing) and can happen at any time after the plugin is initialized.

**When to use:** Always when displaying a plugin's GUI.

**Lifecycle sequence:**
```
1. IEditController::createView(ViewType::kEditor) -> IPlugView*
2. IPlugView::isPlatformTypeSupported("X11EmbedWindowID") -> check result
3. [Host creates parent X11 window via winit]
4. IPlugFrame::new() [host-side COM object providing IRunLoop]
5. IPlugView::setFrame(IPlugFrame*) -> pass host's frame object
6. IPlugView::attached(parent_window_id, "X11EmbedWindowID") -> plugin creates its child window
7. [XEmbed handshake: XEMBED_EMBEDDED_NOTIFY, focus, activation]
8. IPlugView::getSize(&width, &height) -> query plugin's preferred size
9. [Host resizes winit window to match]
10. [Event loop runs: process X11 events, forward to IRunLoop handlers]
11. IPlugView::onSize(new_width, new_height) -> when host window resizes
12. [User closes window]
13. IPlugView::removed() -> plugin destroys its child window
14. IPlugView::setFrame(nullptr) -> disconnect frame
15. IPlugView::release() -> drop COM pointer
```

**Source:** [Steinberg VST3 API Documentation - IPlugView](https://steinbergmedia.github.io/vst3_doc/base/classSteinberg_1_1IPlugView.html), [Steinberg editorhost sample](https://github.com/steinbergmedia/vst3_public_sdk/blob/master/samples/vst-hosting/editorhost/source/platform/linux/window.cpp)

### Pattern 2: XEmbed Protocol for Child Window Embedding

**What:** XEmbed is a standardized protocol for embedding one X11 window inside another. The host (embedder) creates a parent window, the plugin (embedded) creates a child window, and they exchange XEmbed ClientMessage events to coordinate focus, activation, and lifecycle.

**When to use:** Required for all VST3 plugin editor embedding on Linux X11.

**Key atoms (to be interned via `XInternAtom` or `x11rb::atom_manager!`):**
- `_XEMBED` - The XEmbed protocol message type
- `_XEMBED_INFO` - Plugin window property (version, flags)

**XEmbed messages (sent via `XSendEvent` or `x11rb::protocol::xproto::send_event`):**
- `XEMBED_EMBEDDED_NOTIFY` (0) - Host tells plugin "you're embedded in my window"
- `XEMBED_WINDOW_ACTIVATE` (1) - Host window gained focus
- `XEMBED_WINDOW_DEACTIVATE` (2) - Host window lost focus
- `XEMBED_FOCUS_IN` (4) - Plugin should take keyboard focus
- `XEMBED_FOCUS_OUT` (5) - Plugin should release keyboard focus

**Example (using x11rb):**
```rust
use x11rb::atom_manager;
use x11rb::protocol::xproto::*;
use x11rb::connection::Connection;

atom_manager! {
    XEmbedAtoms: XEmbedAtomsCookie {
        _XEMBED,
        _XEMBED_INFO,
    }
}

fn send_xembed_message(
    conn: &impl Connection,
    window: Window,
    message: u32,
    detail: u32,
    data1: u32,
    data2: u32,
) -> Result<(), Box<dyn std::error::Error>> {
    let atoms = XEmbedAtoms::new(conn)?.reply()?;

    let event = ClientMessageEvent {
        response_type: CLIENT_MESSAGE_EVENT,
        format: 32,
        sequence: 0,
        window,
        type_: atoms._XEMBED,
        data: ClientMessageData::from([message, detail, data1, data2, 0]),
    };

    send_event(
        conn,
        false,
        window,
        EventMask::NO_EVENT,
        &event,
    )?;

    conn.flush()?;
    Ok(())
}

// Send XEMBED_EMBEDDED_NOTIFY when plugin window appears
send_xembed_message(conn, plugin_window, 0 /* EMBEDDED_NOTIFY */, 0, timestamp, 0)?;
```

**Source:** [XEmbed Protocol Specification](https://specifications.freedesktop.org/xembed-spec/xembed-spec-latest.html), [Steinberg editorhost Linux implementation](https://github.com/steinbergmedia/vst3_public_sdk/blob/master/samples/vst-hosting/editorhost/source/platform/linux/window.cpp), [yabridge XEmbed usage](https://github.com/robbert-vdh/yabridge)

### Pattern 3: Linux::IRunLoop Host Implementation

**What:** On Linux, there is no global event run loop like on Windows (GetMessage) or macOS (NSRunLoop). VST3 plugins query `Linux::IRunLoop` from the host via `IPlugFrame` (cast from `setFrame()`) to register file descriptor event handlers and timers. The host must provide this interface and dispatch events on the GUI thread's event loop.

**When to use:** Required for all plugins that use UI frameworks (Qt, GTK, VSTGUI) on Linux.

**Interface definition (from VST3 SDK):**
```cpp
namespace Linux {
struct IRunLoop : FUnknown {
    virtual tresult registerEventHandler(
        IEventHandler* handler,
        FileDescriptor fd
    ) = 0;

    virtual tresult unregisterEventHandler(
        IEventHandler* handler
    ) = 0;

    virtual tresult registerTimer(
        ITimerHandler* handler,
        TimerInterval milliseconds
    ) = 0;

    virtual tresult unregisterTimer(
        ITimerHandler* handler
    ) = 0;
};

struct IEventHandler : FUnknown {
    virtual void onFDIsSet(FileDescriptor fd) = 0;
};

struct ITimerHandler : FUnknown {
    virtual void onTimer() = 0;
};
}
```

**Rust implementation strategy:**
```rust
use vst3::Steinberg::Linux::{IRunLoop, IEventHandler, ITimerHandler};
use vst3::com_scrape_types::ComWrapper;
use std::collections::HashMap;
use std::os::unix::io::RawFd;

struct HostRunLoop {
    event_handlers: Mutex<HashMap<RawFd, ComPtr<IEventHandler>>>,
    timers: Mutex<Vec<(ComPtr<ITimerHandler>, std::time::Duration)>>,
}

impl vst3::Class for HostRunLoop {
    type Interfaces = (IRunLoop,);
}

impl IRunLoopTrait for HostRunLoop {
    unsafe fn registerEventHandler(
        &self,
        handler: *mut IEventHandler,
        fd: i32,
    ) -> tresult {
        let handler_ptr = ComPtr::from_raw(handler);
        self.event_handlers.lock().unwrap().insert(fd, handler_ptr);
        kResultOk
    }

    unsafe fn unregisterEventHandler(&self, handler: *mut IEventHandler) -> tresult {
        // Find and remove by handler pointer comparison
        self.event_handlers.lock().unwrap().retain(|_, h| {
            !std::ptr::eq(h.as_ptr(), handler)
        });
        kResultOk
    }

    // Similar for registerTimer/unregisterTimer
}
```

**Integration with winit event loop:**
- Use `mio` or `polling` crate to add FD monitoring to winit's event loop
- On each event loop iteration, check which FDs are ready and call `IEventHandler::onFDIsSet()`
- Use `std::time::Instant` to track timer intervals and call `ITimerHandler::onTimer()` when elapsed

**Source:** [Steinberg VST3 Developer Portal - Provide A Runloop On Linux](https://steinbergmedia.github.io/vst3_dev_portal/pages/Technical+Documentation/Provide+A+Runloop+On+Linux/Index.html), [Steinberg editorhost runloop.h](https://github.com/paulfd/sfizz-vst3/blob/master/public.sdk/samples/vst-hosting/editorhost/source/platform/linux/runloop.h)

### Pattern 4: Threading Model for GUI Window

**What:** The VST3 plugin editor window must run on a dedicated GUI thread separate from the Tokio async runtime (which handles MCP server on stdio). Winit's event loop is blocking and incompatible with Tokio's async event loop.

**When to use:** Always for GUI integration in async Rust applications.

**Architecture:**
```
Thread 1 (main/Tokio):
- MCP server listening on stdio
- AudioHost with Arc<Mutex<PluginInstance>>
- open_editor MCP tool spawns Thread 2

Thread 2 (GUI thread):
- winit::EventLoop::new()
- EditorWindow with IPlugView COM pointer
- HostRunLoop dispatching plugin events
- Blocks in event_loop.run() until window closes
- Signals completion back to Thread 1 via oneshot channel

Communication:
- Thread 1 -> Thread 2: Arc<PluginInstance> passed to GUI thread
- Thread 2 -> Thread 1: Window closed signal via tokio::sync::oneshot
```

**Example:**
```rust
// In AudioHost::open_editor MCP tool
#[tool(description = "Open the plugin's editor window (blocks until window closes)")]
fn open_editor(&self) -> Result<String, String> {
    let plugin_guard = self.plugin.lock().unwrap();
    let plugin = plugin_guard.as_ref()
        .ok_or("No plugin loaded")?;

    // Clone Arc<PluginInstance> to pass to GUI thread
    // (PluginInstance is Send but not Sync -- must be behind Mutex)
    let plugin_arc = Arc::clone(&self.plugin);

    // Spawn GUI thread
    let (tx, rx) = tokio::sync::oneshot::channel();
    std::thread::spawn(move || {
        // GUI thread owns the winit event loop
        let event_loop = winit::event_loop::EventLoop::new();
        let window = EditorWindow::new(&event_loop, plugin_arc);

        event_loop.run(move |event, target| {
            // Process winit events
            // Call IPlugView::onSize on resize
            // Call IPlugView::removed on close
            // Exit loop on window close
            if window_closed {
                target.exit();
            }
        });

        tx.send(()).unwrap(); // Signal completion
    });

    // Wait for window close (blocks MCP tool call until GUI thread exits)
    tokio::task::block_in_place(|| rx.blocking_recv())
        .map_err(|_| "GUI thread panicked")?;

    Ok("editor closed".to_string())
}
```

**Source:** [winit documentation](https://docs.rs/winit/latest/winit/), [Tokio spawn_blocking patterns](https://docs.rs/tokio/latest/tokio/task/fn.spawn_blocking.html)

### Pattern 5: Detecting Plugin's Child Window Creation

**What:** After calling `IPlugView::attached(parent_window_id)`, the plugin creates its own X11 child window inside the parent. The host must detect this window (via `CreateNotify` event) to complete the XEmbed handshake.

**When to use:** Required for XEmbed protocol completion.

**Detection approach:**
```rust
// Select CreateNotify events on parent window
use x11rb::protocol::xproto::*;

create_window(
    conn,
    depth,
    parent_window_id,
    screen.root,
    0, 0, width, height,
    0,
    WindowClass::INPUT_OUTPUT,
    visual_id,
    &CreateWindowAux::new()
        .event_mask(EventMask::SUBSTRUCTURE_NOTIFY | EventMask::FOCUS_CHANGE),
)?;

// In event loop:
match event {
    Event::CreateNotify(create) => {
        if create.parent == parent_window_id {
            let plugin_window = create.window;
            tracing::info!("Plugin created child window: {:x}", plugin_window);

            // Send XEMBED_EMBEDDED_NOTIFY to plugin window
            send_xembed_message(
                conn,
                plugin_window,
                XEMBED_EMBEDDED_NOTIFY,
                0,
                timestamp,
                0,
            )?;

            // Reparent plugin window if needed (some plugins create detached)
            reparent_window(conn, plugin_window, parent_window_id, 0, 0)?;
        }
    }
    // ...
}
```

**Source:** [XEmbed spec section 3.2](https://specifications.freedesktop.org/xembed-spec/xembed-spec-latest.html), [JUCE VST3 Linux implementation discussions](https://forum.juce.com/t/linux-x11-windowing-in-a-vst-3-6-7-plugin-using-juce-components/21767)

### Anti-Patterns to Avoid

- **Running winit event loop on Tokio runtime:** Winit's event loop is blocking and cannot be integrated into Tokio's async event loop. Use a dedicated thread.
- **Calling IPlugView methods from audio thread:** IPlugView is a UI interface. All calls must happen on the GUI thread (the thread that called `attached()`).
- **Not implementing IRunLoop:** Many plugins (especially Qt/GTK based) will fail to display or process events without IRunLoop.
- **Hardcoding X11 Window IDs as integers:** Window IDs are `u32` on X11 but the type varies by platform. Use `raw-window-handle` abstractions.
- **Forgetting to call IPlugView::removed() before closing:** Failing to call `removed()` before dropping the window causes resource leaks and crashes in plugins.
- **Resizing plugin window directly:** The host should resize its own parent window and call `IPlugView::onSize()`. The plugin resizes its own child window.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Cross-platform window creation | Raw X11/Win32/Cocoa FFI | `winit` | Handles event loop differences, DPI scaling, multi-platform quirks; battle-tested by egui/bevy/iced |
| X11 protocol bindings | `xcb` C bindings or manual FFI | `x11rb` | Pure Rust (no C build dependency), type-safe, comprehensive protocol coverage, better error handling |
| Window handle abstraction | Platform-specific types in public API | `raw-window-handle` | Industry standard for window interop; lets us extract X11 Window ID from winit portably |
| Event loop integration | Custom runloop for plugin FDs | `mio` or `polling` crate for FD monitoring | Non-blocking FD polling integrates cleanly with winit's event loop; used by Tokio internally |
| XEmbed message encoding | Manual `XSendEvent` byte packing | `x11rb::protocol::xproto::ClientMessageEvent` | Type-safe message construction, correct padding/endianness, avoids off-by-one errors |
| Timer scheduling | `std::thread::sleep` polling | `std::time::Instant` + event loop integration | Accurate timing without busy-waiting, integrates with IRunLoop::registerTimer |

**Key insight:** This is the first GUI work in the codebase. Use proven libraries (winit for window, x11rb for X11 specifics) rather than building from scratch. The Steinberg C++ sample provides the protocol-level blueprint; we translate it to idiomatic Rust using these libraries.

## Common Pitfalls

### Pitfall 1: IRunLoop Not Provided Causes Silent UI Failures

**What goes wrong:** The plugin's editor window appears but doesn't respond to mouse clicks, or the window is completely black/frozen. No error is reported.

**Why it happens:** Many plugins (Qt-based, GTK-based, VSTGUI on Linux) rely on `Linux::IRunLoop` to register their X11 connection file descriptor for event processing. Without it, the plugin never receives X11 events and can't update its UI.

**How to avoid:**
- Always provide `Linux::IRunLoop` via `IPlugFrame` (cast from `setFrame()`)
- Implement both `registerEventHandler`/`unregisterEventHandler` AND `registerTimer`/`unregisterTimer`
- Integrate FD polling into the winit event loop using `mio` or `polling` crate
- Test with VSTGUI-based plugins (most common on Linux)

**Warning signs:**
- Plugin window appears but is non-interactive
- No error messages in logs
- Plugin works on Windows/macOS but fails on Linux

**Source:** [Steinberg VST3 Developer Portal - Provide A Runloop On Linux](https://steinbergmedia.github.io/vst3_dev_portal/pages/Technical+Documentation/Provide+A+Runloop+On+Linux/Index.html)

### Pitfall 2: XEmbed Handshake Not Completed Causes Detached Windows

**What goes wrong:** The plugin creates a window, but it appears as a separate floating window instead of embedded in the host's window. Or the plugin window appears briefly then disappears.

**Why it happens:** After calling `IPlugView::attached()`, the plugin creates its child window but waits for the `XEMBED_EMBEDDED_NOTIFY` message from the host. If the host doesn't send this message (or sends it to the wrong window ID), the plugin assumes embedding failed and either detaches or closes its window.

**How to avoid:**
- Listen for `CreateNotify` events on the parent window to detect plugin's child window creation
- Send `XEMBED_EMBEDDED_NOTIFY` message immediately after detecting child window
- Verify you're sending to the correct window ID (the child, not the parent)
- Check `_XEMBED_INFO` property on plugin window to verify XEmbed protocol version

**Warning signs:**
- Plugin window floats outside host window
- Plugin window flashes briefly then closes
- XGetWindowProperty for `_XEMBED_INFO` returns `BadWindow` error

**Source:** [XEmbed spec section 3.2 - Notification](https://specifications.freedesktop.org/xembed-spec/xembed-spec-latest.html), [Steinberg editorhost XEmbed implementation](https://github.com/steinbergmedia/vst3_public_sdk/blob/master/samples/vst-hosting/editorhost/source/platform/linux/window.cpp)

### Pitfall 3: Resize Loop Between Host and Plugin

**What goes wrong:** Resizing the host window causes rapid flickering and CPU spikes. The window size oscillates between two values or grows unbounded.

**Why it happens:** The host resizes its window, calls `IPlugView::onSize(new_width, new_height)`, the plugin resizes its child window, the child window sends a `ConfigureNotify` event back to the host, the host interprets this as a size request and calls `onSize()` again, creating a feedback loop.

**How to avoid:**
- The host controls the parent window size; the plugin controls its child window size
- When the user resizes the host window: resize the parent window, then call `IPlugView::onSize()` with the new size
- When `IPlugView::getSize()` returns a new size (plugin requested resize): resize the parent window to match, do NOT call `onSize()` again
- Ignore `ConfigureNotify` events from the plugin's child window (they're internal to the plugin)
- Add a "resize in progress" flag to prevent re-entrant `onSize()` calls

**Warning signs:**
- Window flickers rapidly when resizing
- CPU usage spikes to 100% during resize
- Window size jumps between two values
- Logs show repeated `onSize()` calls with alternating dimensions

**Source:** [JUCE VST3 resize fix for Ardour](https://github.com/juce-framework/JUCE/pull/867), [Ardour VST3 resize loop prevention](https://github.com/Ardour/ardour/pull/599)

### Pitfall 4: Platform Type String Mismatch

**What goes wrong:** `IPlugView::attached()` returns an error (kInvalidArgument or kResultFalse), and the editor never appears.

**Why it happens:** The plugin expects a specific platform type string, but the host passes a different one. On Linux, the standard is `"X11EmbedWindowID"`, but some documentation shows `"kPlatformTypeX11EmbedWindowID"` (with `k` prefix), which is the C++ constant name, not the actual string.

**How to avoid:**
- Always pass `"X11EmbedWindowID"` as the platform type string (no `k` prefix)
- Call `IPlugView::isPlatformTypeSupported("X11EmbedWindowID")` before `attached()` to verify support
- Check the return value of `attached()` -- if it fails, log the error and try fallback approaches (some plugins accept raw Window IDs without XEmbed)

**Warning signs:**
- `attached()` returns non-zero error code
- `isPlatformTypeSupported()` returns false for `"X11EmbedWindowID"`
- Plugin works on other hosts (REAPER, Bitwig) but not your host

**Source:** [VST3 Host - how to display plugin GUI? - KVR Forum](https://www.kvraudio.com/forum/viewtopic.php?t=334466), [Steinberg IPlugView documentation](https://steinbergmedia.github.io/vst3_doc/base/classSteinberg_1_1IPlugView.html)

### Pitfall 5: Calling GUI Methods from Wrong Thread

**What goes wrong:** Random crashes, segfaults, or "BadWindow" X11 errors. The errors are non-deterministic and hard to reproduce.

**Why it happens:** X11 (and COM on Windows) is not thread-safe. All IPlugView methods (`attached`, `removed`, `onSize`, etc.) must be called from the same thread that created the window (the GUI thread). If the MCP server thread or audio processing thread calls these methods, it corrupts X11 state.

**How to avoid:**
- All IPlugView interaction happens on the dedicated GUI thread (Thread 2 in Pattern 4)
- The MCP `open_editor` tool blocks until the GUI thread exits (use `tokio::sync::oneshot` for signaling)
- Never call `IPlugView` methods from `PluginInstance::process()` (audio thread)
- Use message passing (channels) if the MCP server needs to send commands to the GUI thread

**Warning signs:**
- Crashes with X11 error messages like "BadWindow" or "BadDrawable"
- Segfaults in `XSendEvent` or plugin UI framework code
- Works sometimes, crashes other times (race condition)
- Plugin works in other hosts but crashes in your host

**Source:** Established X11/GUI threading best practices, VST3 SDK documentation (IPlugView is UI-thread only)

## Code Examples

### Example 1: Opening Editor Window (MCP Tool)

```rust
// In server.rs AudioHost

#[tool(description = "Open the plugin's graphical editor window (blocks until closed)")]
fn open_editor(&self) -> Result<String, String> {
    info!("open_editor called");

    // Check that a plugin is loaded
    let plugin_guard = self.plugin.lock()
        .map_err(|e| format!("Lock error: {}", e))?;

    if plugin_guard.is_none() {
        return Err("No plugin loaded. Call load_plugin first.".to_string());
    }
    drop(plugin_guard); // Release lock before spawning thread

    // Clone Arc references for GUI thread
    let plugin_arc = Arc::clone(&self.plugin);
    let plugin_info_arc = Arc::clone(&self.plugin_info);

    // Channel to signal window close
    let (tx, rx) = tokio::sync::oneshot::channel();

    // Spawn GUI thread (must be separate from Tokio runtime)
    std::thread::spawn(move || {
        match gui::open_editor_window(plugin_arc, plugin_info_arc) {
            Ok(_) => {
                info!("Editor window closed normally");
                let _ = tx.send(Ok(()));
            }
            Err(e) => {
                error!("Editor window failed: {}", e);
                let _ = tx.send(Err(format!("Editor error: {}", e)));
            }
        }
    });

    // Block until window closes (blocks MCP tool call)
    tokio::task::block_in_place(|| {
        rx.blocking_recv()
            .map_err(|_| "GUI thread panicked".to_string())?
    })?;

    Ok("Editor closed".to_string())
}
```

**Source:** Adapted from Pattern 4 threading model.

### Example 2: EditorWindow Struct (Winit + IPlugView)

```rust
// In src/gui/window.rs

use winit::event_loop::EventLoop;
use winit::window::{Window, WindowBuilder};
use winit::platform::unix::WindowExtUnix; // For x11_window()
use vst3::com_scrape_types::ComPtr;
use vst3::Steinberg::IPlugView;
use raw_window_handle::{HasRawWindowHandle, RawWindowHandle};

pub struct EditorWindow {
    window: Window,
    plug_view: ComPtr<IPlugView>,
    plugin_window_id: Option<u32>, // X11 Window ID of plugin's child window
}

impl EditorWindow {
    pub fn new(
        event_loop: &EventLoop<()>,
        plugin: Arc<Mutex<Option<PluginInstance>>>,
    ) -> Result<Self, Box<dyn std::error::Error>> {
        // Lock plugin to access controller
        let plugin_guard = plugin.lock().unwrap();
        let plugin_inst = plugin_guard.as_ref()
            .ok_or("No plugin loaded")?;

        // Get IEditController and create view
        let controller = plugin_inst.controller()
            .ok_or("Plugin has no edit controller")?;

        let plug_view: ComPtr<IPlugView> = unsafe {
            let mut view_ptr: *mut c_void = std::ptr::null_mut();
            let result = controller.createView(
                b"editor\0".as_ptr() as *const i8, // ViewType::kEditor
            );
            if result.is_null() {
                return Err("createView returned null".into());
            }
            ComPtr::from_raw(result).ok_or("Invalid IPlugView pointer")?
        };

        // Check platform support
        let platform_type = "X11EmbedWindowID";
        let supported = unsafe {
            plug_view.isPlatformTypeSupported(
                platform_type.as_ptr() as *const i8
            )
        };
        if supported != vst3::Steinberg::kResultOk {
            return Err(format!("Plugin doesn't support {}", platform_type).into());
        }

        // Query plugin's preferred size
        let mut width: i32 = 800;
        let mut height: i32 = 600;
        unsafe {
            if plug_view.getSize(&mut width, &mut height) != vst3::Steinberg::kResultOk {
                tracing::warn!("getSize failed, using default 800x600");
            }
        }

        // Create winit window
        let window = WindowBuilder::new()
            .with_title("Plugin Editor")
            .with_inner_size(winit::dpi::PhysicalSize::new(width as u32, height as u32))
            .with_resizable(true)
            .build(event_loop)?;

        // Get X11 Window ID from winit
        let parent_window_id = match window.raw_window_handle() {
            RawWindowHandle::Xlib(handle) => handle.window as u32,
            RawWindowHandle::Xcb(handle) => handle.window,
            _ => return Err("Not running on X11".into()),
        };

        // Create and set IPlugFrame (provides IRunLoop)
        let plug_frame = gui::plugframe::PlugFrame::new()?;
        unsafe {
            plug_view.setFrame(
                plug_frame.to_com_ptr::<vst3::Steinberg::IPlugFrame>()
                    .ok_or("Failed to get IPlugFrame COM pointer")?
                    .as_ptr()
            );
        }

        // Attach plugin view to window
        unsafe {
            let result = plug_view.attached(
                parent_window_id as *mut c_void,
                platform_type.as_ptr() as *const i8,
            );
            if result != vst3::Steinberg::kResultOk {
                return Err(format!("attached() failed: {}", result).into());
            }
        }

        tracing::info!(
            "Editor attached to X11 window {:x} ({}x{})",
            parent_window_id, width, height
        );

        Ok(EditorWindow {
            window,
            plug_view,
            plugin_window_id: None, // Detected later via CreateNotify
        })
    }

    pub fn handle_resize(&mut self, new_width: u32, new_height: u32) {
        unsafe {
            let result = self.plug_view.onSize(new_width as i32, new_height as i32);
            if result != vst3::Steinberg::kResultOk {
                tracing::warn!("onSize failed: {}", result);
            }
        }
    }
}

impl Drop for EditorWindow {
    fn drop(&mut self) {
        // Detach plugin view before dropping window
        unsafe {
            let result = self.plug_view.removed();
            if result != vst3::Steinberg::kResultOk {
                tracing::warn!("removed() failed: {}", result);
            }

            // Clear frame reference
            self.plug_view.setFrame(std::ptr::null_mut());
        }

        tracing::info!("Editor window dropped");
    }
}
```

// In src/gui/runloop.rs

use vst3::Steinberg::Linux::{IRunLoop, IEventHandler, ITimerHandler};
use vst3::com_scrape_types::{ComWrapper, ComPtr};
use std::collections::HashMap;
use std::sync::Mutex;
use std::os::unix::io::RawFd;

pub struct HostRunLoop {
    event_handlers: Mutex<HashMap<RawFd, ComPtr<IEventHandler>>>,
    timers: Mutex<Vec<TimerEntry>>,
}

struct TimerEntry {
    handler: ComPtr<ITimerHandler>,
    interval_ms: u64,
    last_fire: std::time::Instant,
}

impl HostRunLoop {
    pub fn new() -> ComWrapper<Self> {
        ComWrapper::new(HostRunLoop {
            event_handlers: Mutex::new(HashMap::new()),
            timers: Mutex::new(Vec::new()),
        })
    }

    /// Called from event loop when FDs are ready (via mio/polling).
    pub fn dispatch_ready_fds(&self, ready_fds: &[RawFd]) {
        let handlers = self.event_handlers.lock().unwrap();
        for &fd in ready_fds {
            if let Some(handler) = handlers.get(&fd) {
                unsafe { handler.onFDIsSet(fd) };
            }
        }
    }

    /// Called from event loop on each iteration to fire timers.
    pub fn dispatch_timers(&self) {
        let mut timers = self.timers.lock().unwrap();
        let now = std::time::Instant::now();

        for entry in timers.iter_mut() {
            let elapsed = now.duration_since(entry.last_fire).as_millis() as u64;
            if elapsed >= entry.interval_ms {
                unsafe { entry.handler.onTimer() };
                entry.last_fire = now;
            }
        }
    }
}

impl vst3::Class for HostRunLoop {
    type Interfaces = (IRunLoop,);
}

impl vst3::Steinberg::Linux::IRunLoopTrait for HostRunLoop {
    unsafe fn registerEventHandler(
        &self,
        handler: *mut IEventHandler,
        fd: vst3::Steinberg::Linux::FileDescriptor,
    ) -> vst3::Steinberg::tresult {
        let handler_ptr = ComPtr::from_raw(handler)
            .ok_or(vst3::Steinberg::kInvalidArgument)
            .unwrap();

        self.event_handlers.lock().unwrap().insert(fd, handler_ptr);
        tracing::debug!("Registered event handler for FD {}", fd);
        vst3::Steinberg::kResultOk
    }

    unsafe fn unregisterEventHandler(
        &self,
        handler: *mut IEventHandler,
    ) -> vst3::Steinberg::tresult {
        self.event_handlers.lock().unwrap().retain(|_, h| {
            !std::ptr::eq(h.as_ptr(), handler)
        });
        tracing::debug!("Unregistered event handler");
        vst3::Steinberg::kResultOk
    }

    unsafe fn registerTimer(
        &self,
        handler: *mut ITimerHandler,
        interval_ms: vst3::Steinberg::Linux::TimerInterval,
    ) -> vst3::Steinberg::tresult {
        let handler_ptr = ComPtr::from_raw(handler)
            .ok_or(vst3::Steinberg::kInvalidArgument)
            .unwrap();

        self.timers.lock().unwrap().push(TimerEntry {
            handler: handler_ptr,
            interval_ms: interval_ms as u64,
            last_fire: std::time::Instant::now(),
        });
        tracing::debug!("Registered timer with interval {}ms", interval_ms);
        vst3::Steinberg::kResultOk
    }

    unsafe fn unregisterTimer(
        &self,
        handler: *mut ITimerHandler,
    ) -> vst3::Steinberg::tresult {
        self.timers.lock().unwrap().retain(|entry| {
            !std::ptr::eq(entry.handler.as_ptr(), handler)
        });
        tracing::debug!("Unregistered timer");
        vst3::Steinberg::kResultOk
    }
}

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Custom window libraries per platform | `winit` as standard Rust cross-platform windowing | winit 0.30 (2024) | Unified API, well-maintained, used by major frameworks (egui, bevy, iced) |
| `xcb` C bindings for X11 | `x11rb` pure Rust X11 bindings | x11rb 0.10+ (2021-2023) | No C dependencies, better error handling, comprehensive protocol support |
| `raw-window-handle` 0.5 | `raw-window-handle` 0.6 (2024) | Breaking changes in API, more platforms | Safer FFI, supports more window systems |
| XEmbed as optional | XEmbed required for VST3 on Linux | VST3 spec established ~2018 | Standard protocol for all hosts (REAPER, Bitwig, Ardour) |
| IRunLoop optional on Linux | IRunLoop required for many plugins | Documented in VST3 SDK (ongoing) | Qt/GTK/VSTGUI plugins fail without it |

**Deprecated/outdated:**
- **Direct xcb/xlib bindings:** Use `x11rb` instead (pure Rust, safer, better maintained).
- **Ignoring IRunLoop:** Modern plugins (2020+) expect hosts to provide IRunLoop on Linux.
- **Assuming platform type "kPlatformTypeX11EmbedWindowID":** The string is `"X11EmbedWindowID"` (no `k` prefix).

## Open Questions

1. **Which plugins should we test with during development?**
   - What we know: Need both VSTGUI-based and Qt/GTK-based plugins for coverage
   - What's unclear: Which free/open-source Linux VST3 plugins provide good test coverage
   - Recommendation: Surge XT (VSTGUI, open source, complex UI), u-he Protoverb (free, Qt-based), Sfizz (open source, VSTGUI)

2. **Should we support resizing, or fixed-size window only?**
   - What we know: Resizing requires careful handling to avoid loops (Pitfall 3)
   - What's unclear: Whether offline MVP needs resizable editor, or if fixed-size is acceptable for Phase 04.1
   - Recommendation: Start with fixed-size (simpler, avoids resize pitfalls), add resizing in follow-up if users request it

3. **How to handle plugins that don't support X11EmbedWindowID?**
   - What we know: Most modern plugins support XEmbed, but some old plugins may not
   - What's unclear: Fallback strategy (fail gracefully, or try raw Window ID without XEmbed?)
   - Recommendation: Log error and fail gracefully in Phase 04.1; add fallback in later phase if needed

4. **Should open_editor block or return immediately?**
   - What we know: MCP tool call blocks until window closes (Pattern 4 example)
   - What's unclear: Whether AI agent expects immediate return (window opens in background) or blocking (window must close before next command)
   - Recommendation: Blocking is simpler for Phase 04.1 (AI opens editor, interacts, closes editor, continues); non-blocking requires separate "close_editor" tool

5. **How to integrate FD polling with winit event loop?**
   - What we know: `mio` or `polling` crate can monitor FDs, winit event loop is platform-specific
   - What's unclear: Best approach for integrating FD polling into winit's event loop without platform-specific code paths
   - Recommendation: Use `mio` (Tokio's internal dependency, well-tested) with custom event source; alternatively, poll FDs on each event loop iteration (simpler but less efficient)
