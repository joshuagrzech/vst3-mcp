    **1. Update `src/lib.rs`:**

    **2. Create `src/audio/mod.rs`:**
    Re-export: `pub mod decode; pub mod encode; pub mod buffers; pub mod process;`

    **3. Create `src/audio/decode.rs`:**
    Implement multi-format audio decoding using symphonia (see research code example):
    - Define `DecodedAudio { samples: Vec<f32>, channels: usize, sample_rate: u32, total_frames: usize }` where samples are INTERLEAVED f32.
    - `pub fn decode_audio_file(path: &Path) -> Result<DecodedAudio>`:
      Open file, create `MediaSourceStream`, probe format with hint from file extension.
      Get default track, extract channels and sample_rate from codec_params.
      Create decoder, loop over packets: decode each packet, use `SampleBuffer::<f32>::new()` + `copy_interleaved_ref()` to convert to f32, extend the samples Vec.
      Handle EOF gracefully (match on `UnexpectedEof` IoError).
      Return DecodedAudio with computed total_frames = samples.len() / channels.
    - **Error handling:** Return `anyhow::Error` with context messages. If no audio track found, return descriptive error.

    **4. Create `src/audio/encode.rs`:**
    Implement WAV output using hound (see research code example):
    - `pub fn write_wav(path: &Path, samples: &[f32], channels: u16, sample_rate: u32) -> Result<()>`:
      Create `WavSpec { channels, sample_rate, bits_per_sample: 32, sample_format: SampleFormat::Float }`.
      Create `WavWriter`, write each sample, call `finalize()`.
    - This preserves full 32-bit float precision. No dithering or bit depth conversion needed.

    **5. Create `src/audio/buffers.rs`:**
    Implement interleaved <-> deinterleaved conversion (critical for VST3 compatibility):
    - `pub fn deinterleave(interleaved: &[f32], channels: usize) -> Vec<Vec<f32>>`:
      Convert interleaved samples (L R L R L R) to per-channel vectors ([L L L], [R R R]).
      Total frames = interleaved.len() / channels.
      For each frame, distribute samples to the corresponding channel vector.
    - `pub fn interleave(planar: &[Vec<f32>]) -> Vec<f32>`:
      Convert per-channel vectors back to interleaved.
      Iterate frame by frame, push each channel's sample.
    - **Unit tests (REQUIRED):**
      - Round-trip test: interleave(deinterleave(data)) == data
      - Known signal test: 2-channel [1,2,3,4,5,6] deinterleaves to [[1,3,5], [2,4,6]]
      - Single channel: deinterleave with channels=1 returns one vector unchanged
      - Empty input: handle gracefully

    **6. Create `src/audio/process.rs`:**
    Implement offline block-based rendering:
    - `pub fn render_offline(plugin: &mut PluginInstance, decoded: &DecodedAudio) -> Result<Vec<f32>>`:
      1. Deinterleave input audio using `buffers::deinterleave()`.
      2. Determine max_block_size from the plugin's setup (use 4096 as default if not queryable).
      3. Allocate output channel buffers (same channel count and total_frames as input, plus tail room).
      4. Process in blocks:
         ```
         while offset < total_frames:
           block_size = min(max_block_size, total_frames - offset)
           build per-channel input slices [offset..offset+block_size]
           build per-channel output slices [offset..offset+block_size]
           call plugin.process(input_slices, output_slices, block_size)
           offset += block_size
         ```
      5. **Tail handling:** After all input is processed, query `plugin.get_tail_samples()` (via `IAudioProcessor::getTailSamples()`). Feed silence for that many additional samples (in blocks). If tail is `kInfiniteTail`, use a configurable max of 30 seconds worth of samples at the given sample rate. Append tail output to the result.
      6. Interleave output channels using `buffers::interleave()`.
      7. Return the interleaved f32 output samples.
    - **Anti-patterns to avoid:**
      - Do NOT allocate inside the block loop. Pre-allocate all buffers before the loop starts.
      - Do NOT pass interleaved data to the plugin. VST3 expects planar (one pointer per channel).
  </action>

  <action>
    **1. Create `src/server.rs`:**
    Implement the MCP server using rmcp macros (see research Pattern 3):

    - **AudioHost struct:**
      ```rust
      pub struct AudioHost {
          plugin: Arc<Mutex<Option<PluginInstance>>>,
          plugin_info: Arc<Mutex<Option<PluginInfo>>>,
      }
      ```
      Use `Arc<Mutex<>>` because rmcp requires `Send + Sync` for the handler. The Mutex protects single-threaded plugin access. Only ONE plugin loaded at a time (Phase 1 scope).

    - **Tool input structs** (all derive `Debug, Deserialize, JsonSchema`):
      - `ScanPluginsRequest { path: Option<String> }` -- optional custom scan directory
      - `LoadPluginRequest { uid: String, sample_rate: Option<u32> }` -- plugin UID from scan results, optional sample rate override
      - `ProcessAudioRequest { input_file: String, output_file: String }` -- input/output file paths
      - `SavePresetRequest { path: String }` -- output .vstpreset file path
      - `LoadPresetRequest { path: String }` -- input .vstpreset file path
      Add `#[schemars(description = "...")]` on each field for clear tool documentation.

    - **Tool implementations** (`#[tool_router] impl AudioHost`):

      - `scan_plugins`: Call `scanner::scan_plugins()` inside `spawn_blocking`. Return JSON array of PluginInfo. Include uid, name, vendor, category for each plugin.

      - `load_plugin`:
        1. Run scan to find the plugin by UID (or use cached scan results)
        2. Load the module via `VstModule::load(plugin_path)`
        3. Create `PluginInstance::from_factory()` with the class ID
        4. Call `setup()` with sample_rate (default 44100 if not specified, will be overridden by input file rate during processing)
        5. Call `activate()` and `start_processing()`
        6. Store in `self.plugin`
        7. Return success with plugin name, parameter count, bus info

      - `process_audio`:
        1. Check that a plugin is loaded (return error if not)
        2. Inside `spawn_blocking`:
           a. Decode input file via `audio::decode::decode_audio_file()`
           b. If plugin was set up with different sample rate than input, re-setup with input's sample rate (stop_processing -> deactivate -> setup(new_rate) -> activate -> start_processing)
           c. Call `audio::process::render_offline()` with the plugin and decoded audio
           d. Write output via `audio::encode::write_wav()` using input's sample_rate and channels
        3. Return success with output file path, duration, sample rate, channels

      - `save_preset`:
        1. Check plugin loaded
        2. Inside `spawn_blocking`: call `preset::state::save_plugin_state()`
        3. Return success with file path

      - `load_preset`:
        1. Check plugin loaded
        2. Inside `spawn_blocking`: call `preset::state::restore_plugin_state()`
        3. Return success with confirmation

    - **Error handling pattern:** All tools return `Result<CallToolResult, McpError>`. Convert internal errors to `McpError::internal(error_message)`. Return user-friendly error messages (e.g., "No plugin loaded. Call load_plugin first." not a raw unwrap panic).

    - **ServerHandler implementation:**
      ```rust
      #[tool_handler]
      impl ServerHandler for AudioHost {
          fn get_info(&self) -> ServerInfo {
              ServerInfo {
                  instructions: Some("VST3 audio processing host. Scan for plugins, load one, process audio files, and manage presets.".into()),
                  capabilities: ServerCapabilities::builder()
                      .enable_tools()
                      .build(),
                  ..Default::default()
              }
          }
      }
      ```

    **2. Create `src/main.rs`:**
    - Initialize tracing subscriber with env filter (RUST_LOG support):
      ```rust
      tracing_subscriber::fmt()
          .with_env_filter(tracing_subscriber::EnvFilter::from_default_env())
          .with_writer(std::io::stderr)  // MCP uses stdout for protocol, logs go to stderr
          .init();
      ```
    - **Critical:** All tracing/logging output MUST go to stderr. stdout is reserved for MCP JSON-RPC protocol messages. Writing anything non-protocol to stdout will break MCP communication.
    - Create `AudioHost::new()` and start the MCP server:
      ```rust
      #[tokio::main]
      async fn main() -> Result<(), Box<dyn std::error::Error>> {
          // init tracing to stderr
          let host = AudioHost::new();
          let service = host.serve(rmcp::transport::stdio()).await?;
          service.waiting().await?;
          Ok(())
      }
      ```
    - Verify the exact rmcp API for starting a stdio server. The research shows `host.serve(stdio()).await?` but the import path may differ. Check rmcp docs/examples for the current API.

    **3. Thread safety note:**
    VST3 plugins have thread affinity -- all COM calls for a single plugin instance should happen on the same thread. The `spawn_blocking` approach works because tokio's blocking pool reuses threads, and the Mutex ensures only one operation at a time. However, if issues arise, consider using a dedicated thread with a channel-based command pattern instead of Mutex + spawn_blocking.
