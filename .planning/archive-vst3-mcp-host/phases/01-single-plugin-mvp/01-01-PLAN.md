    **2. Create `src/lib.rs`:**
    Declare public modules: `pub mod hosting;` and `pub mod preset;`. (Audio module added in Plan 01-02.)

    **3. Create `src/hosting/types.rs`:**
    Define shared types used across the hosting layer:
    - `PluginInfo`: name (String), vendor (String), uid (String -- hex-encoded class ID), category (String), version (String), path (PathBuf). Derive `Debug, Clone, Serialize, Deserialize`.
    - `BusInfo`: name (String), channel_count (i32), bus_type (enum: Audio/Event), direction (enum: Input/Output), is_default_active (bool). Derive `Debug, Clone`.
    - `ParamInfo`: id (u32), title (String), units (String), default_normalized (f64), step_count (i32), flags (u32). Derive `Debug, Clone`.
    - `PluginState` enum: `Created`, `SetupDone`, `Active`, `Processing`. Derive `Debug, Clone, Copy, PartialEq`.
    - Error type `HostError` using thiserror with variants: `ModuleLoadFailed(String)`, `FactoryError(String)`, `InitializeFailed(String)`, `SetupFailed(String)`, `ActivationFailed(String)`, `ProcessingFailed(String)`, `PresetError(String)`, `ScanError(String)`.

    **4. Create `src/hosting/module.rs`:**
    Implement `VstModule` -- a safe wrapper around loading a .vst3 bundle's shared library:
    - Use `libloading` crate (`cargo add libloading`) to dlopen the platform-specific binary inside the .vst3 bundle directory:
      - Linux: `Contents/x86_64-linux/MODULE_NAME.so`
      - macOS: `Contents/MacOS/MODULE_NAME`
      - Windows: `Contents/x86_64-win/MODULE_NAME.dll`
    - Look up and call the VST3 module entry function. The vst3 crate should provide the factory entry point signature. The standard C entry points are `GetPluginFactory` (returns `IPluginFactory*`), `InitDll`/`ExitDll` (module lifecycle).
    - Store the loaded library handle and the `ComPtr<IPluginFactory>` as fields.
    - Implement `Drop` to call `ExitDll` before unloading the library.
    - Expose `pub fn factory(&self) -> &ComPtr<IPluginFactory>` for plugin instantiation.
    - **Important:** Study the exact vst3 crate API during implementation. The crate may provide a `Module` type or helper for loading. If so, use it instead of raw libloading. The research flagged this as LOW confidence -- adapt to what the crate actually provides.

    **5. Create `src/hosting/scanner.rs`:**
    Implement plugin scanning:
    - `pub fn default_scan_paths() -> Vec<PathBuf>`: Return OS-specific VST3 plugin directories (see research code example for Linux/macOS/Windows paths).
    - `pub fn scan_plugins(custom_path: Option<&str>) -> Result<Vec<PluginInfo>>`: Walk scan directories recursively, find `.vst3` directories/bundles. For each bundle:
      1. **Fast path:** Try reading `Contents/Resources/moduleinfo.json`. Parse JSON to extract class info (name, vendor, uid, category, version) without loading the binary. This is the Steinberg-standard metadata file.
      2. **Slow path (fallback):** If no moduleinfo.json, load the module via `VstModule`, query `IPluginFactory`, iterate class info with `countClasses()`/`getClassInfo()` or `getClassInfo2()`, extract metadata. Unload module after scanning.
      3. Build `PluginInfo` for each audio processor class (filter by category containing "Audio" or subcategory "Fx" or "Instrument").
    - Handle errors gracefully: log warnings for unparseable plugins, skip them, continue scanning.

    **6. Create `src/hosting/mod.rs`:**
    Re-export: `pub mod types; pub mod module; pub mod scanner; pub mod plugin; pub mod host_app;`
    (plugin and host_app modules will have stub files until Task 2 fills them.)

    **Verify compilation:** `cargo check` must pass with no errors. Create minimal stubs for plugin.rs and host_app.rs if needed to satisfy the mod.rs declarations.

  <action>
    **1. Create `src/hosting/host_app.rs`:**
    Implement the host-side COM interfaces that plugins require:

    - **`HostApp`**: Implement `IHostApplication` COM interface using the vst3 crate's `Class` trait (or equivalent mechanism). This provides:
      - `getName()`: Return the host name (e.g., "VST3 MCP Host")
      - `createInstance()`: Return `kNotImplemented` for Phase 1 (no message passing needed yet)
    - **`ComponentHandler`**: Implement `IComponentHandler` COM interface. This handles callbacks from the plugin's edit controller:
      - `beginEdit(id)` / `performEdit(id, value)` / `endEdit(id)`: Log parameter changes. In Phase 1, these can be no-ops with tracing::debug logging.
      - `restartComponent(flags)`: Log restart requests. Handle `kLatencyChanged` and `kParamValuesChanged` flags minimally.
    - Both implementations must be wrapped in `ComWrapper` (or equivalent from vst3 crate) to produce proper COM pointers that can be passed to plugins.
    - **Important:** The exact mechanism for implementing COM interfaces as a host in the vst3 crate is flagged as LOW confidence in research. Study the crate's `Class` trait, `ComWrapper`, and any host-side examples. Adapt the implementation to match what the crate actually provides. If the crate does not support host-side COM implementation, consider using raw vtable construction with `#[repr(C)]` structs.

    **2. Create `src/hosting/plugin.rs`:**
    Implement `PluginInstance` -- the core plugin lifecycle manager with state machine enforcement:

    - **Construction** (`pub fn from_factory(factory, class_id, host) -> Result<Self>`):
      1. Call `factory.createInstance(class_id, IComponent::IID)` to create the component
      2. Call `component.initialize(host_context)` passing the HostApp as `FUnknown`
      3. Query `IAudioProcessor` from the component via `cast()` or `query_interface()`
      4. Query or create `IEditController`:
         - Try `component.query_interface::<IEditController>()`
         - If that fails, get controller class ID via `IComponent::getControllerClassId()`, create from factory
         - Call `controller.initialize(host_context)`
      5. Set the `IComponentHandler` on the controller via `controller.setComponentHandler(handler)`
      6. Connect component <-> controller via `IConnectionPoint` (query from both, call `connect()`)
      7. Return `PluginInstance` in `Created` state

    - **State transitions** (enforce via methods that consume self and return new state):
      - `setup(self, sample_rate: f64, max_block_size: i32) -> Result<Self>`:
        Call `processor.setupProcessing()` with `ProcessSetup { processMode: kOffline, symbolicSampleSize: kSample32, sampleRate, maxSamplesPerBlock }`.
        Call `component.activateBus()` for default audio input/output buses.
        Transition state to `SetupDone`.
      - `activate(self) -> Result<Self>`: Call `component.setActive(1)`. Transition to `Active`.
      - `start_processing(self) -> Result<Self>`: Call `processor.setProcessing(1)`. Transition to `Processing`.
      - `stop_processing(self) -> Result<Self>`: Call `processor.setProcessing(0)`. Transition to `Active`.
      - `deactivate(self) -> Result<Self>`: Call `component.setActive(0)`. Transition to `SetupDone`.

    - **Processing** (only available in Processing state):
      - `pub fn process(&mut self, inputs: &[&[f32]], outputs: &mut [&mut [f32]], num_samples: i32) -> Result<()>`:
        Build `ProcessData` struct with audio buffers, call `processor.process(&mut process_data)`.
        This is the most complex unsafe code -- carefully construct the `AudioBusBuffers` and `ProcessData` structs matching the VST3 C ABI layout.

    - **Parameter access:**
      - `pub fn get_parameter_count(&self) -> i32`: Via `controller.getParameterCount()`
      - `pub fn get_parameter_info(&self, index: i32) -> Result<ParamInfo>`: Via `controller.getParameterInfo()`
      - `pub fn get_parameter(&self, id: u32) -> f64`: Via `controller.getParamNormalized()`
      - `pub fn queue_parameter_change(&mut self, id: u32, value: f64)`: Queue for next process() call
      - Internal: Deliver queued parameter changes via `IParameterChanges` in ProcessData

    - **Bus info:**
      - `pub fn get_bus_info(&self) -> Vec<BusInfo>`: Query input/output bus info from component

    - **Drop implementation:** Ensure teardown order: setProcessing(false) -> setActive(false) -> terminate component -> terminate controller. ComPtr handles Release.

    - **Design note:** The research suggests encoding states as separate Rust types (Created, SetupDone, Active, Processing) for compile-time enforcement. If this proves too cumbersome with the actual vst3 crate API (e.g., multiple COM pointers need to move between states), use a runtime enum approach with PluginState checks instead. Prioritize working code over type-level purity.

    **3. Create `src/preset/vstpreset.rs`:**
    Implement .vstpreset binary format read/write:
    - `pub fn save_preset(path: &Path, class_id: &[u8; 32], component_state: &[u8], controller_state: Option<&[u8]>) -> Result<()>`:
      Write 48-byte header (magic "VST3", version 1, class_id ASCII, chunk_list_offset placeholder).
      Write component state chunk data. Optionally write controller state chunk data.
      Write chunk list at end: "List" magic, count, entries with chunk IDs ("Comp"/"Cont"), offsets, sizes.
      Seek back and patch chunk_list_offset in header.
    - `pub fn load_preset(path: &Path) -> Result<PresetData>`:
      Read and validate header (check "VST3" magic, version).
      Seek to chunk list, read entries.
      Read component state chunk ("Comp") and controller state chunk ("Cont") by offset/size.
      Return `PresetData { class_id, component_state: Vec<u8>, controller_state: Option<Vec<u8>> }`.
    - All integers are little-endian. Use `std::io::{Read, Write, Seek}` with `byteorder` crate or `i32::to_le_bytes()`.

    **4. Create `src/preset/state.rs`:**
    Bridge between plugin instances and .vstpreset files:
    - `pub fn save_plugin_state(plugin: &PluginInstance, path: &Path) -> Result<()>`:
      Call `component.getState()` to get processor state bytes.
      Call `controller.getState()` to get controller state bytes (optional -- some plugins don't have separate controller state).
      Get class ID from plugin. Call `vstpreset::save_preset()`.
    - `pub fn restore_plugin_state(plugin: &mut PluginInstance, path: &Path) -> Result<()>`:
      Call `vstpreset::load_preset()` to get PresetData.
      Validate class_id matches the loaded plugin.
      Call `component.setState()` with component state.
      Call `controller.setComponentState()` with the SAME component state (required by spec to sync controller).
      If controller state exists, call `controller.setState()` with it.
    - **Important (Pitfall 6):** After component.setState(), ALWAYS call controller.setComponentState() with the same data. This is a common mistake -- the spec requires both calls to keep component and controller in sync.

    **5. Create `src/preset/mod.rs`:**
    Re-export: `pub mod vstpreset; pub mod state;`

    **6. Update `src/lib.rs`:**
    Add `pub mod preset;` if not already present.

    **Implementation note on getState/setState streams:** VST3's getState/setState use `IBStream` COM interface. You'll need to implement a minimal `IBStream` backed by a `Vec<u8>` (or `Cursor<Vec<u8>>`). Implement `read`, `write`, `seek`, `tell` methods. Use the vst3 crate's COM interface implementation pattern (same as HostApp).
